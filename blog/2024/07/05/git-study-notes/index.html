    Git Study Notes

    Commands¶

    [!error] WARNING 注意带--参数的**空格**

    Basic Commands¶

    1. git add xxx.md
    2. git add . 添加全部
    3. git diff 查看区别
    4. git commit -m "comment" 提交并添加备注
    5. git status 查看当前状态

    Rollback¶

    1. git log 查看**完整**日志
    2. git log --pretty=oneline 简要日志，只一行，显示当前的版本号和commit信息
    3. git reset --hard xxxx 参数
      • HEAD表示当前版本
      • HEAD^, HEAD^^, HEAD~100 前一个，前两个，前100。只能回退
      • commit id 不需要完整的id，显示出差异化即可。可以在回退后再回去之后的版本
    4. git reflog 记录每一次命令和差异化的commit id便于吃后悔药 back to future
    5. git checkout -- xx.md 尚未进行git add 添加到工作区时，丢弃工作区的修改
    6. git reset HEAD readme.txt 在git add进入暂存区后将暂存区的修改撤销，重放回工作区

    Remove¶

    本地删除rm readme.md ,此时工作区和版本库不一致，git status会有所显示 接下来 1. 确实要删git rm readme.md 从版本库中删除，然后git commit 2. 删错了git checkout readme.md 用版本库里的版本替换工作区的版本

    Concepts and Principles¶

    • .git 隐藏目录，是版本库
    • stage or index 是暂存区，在版本库中
    • git add进入暂存区，git commit提交进分支
    • Git自动创建的分支为master
    • git commit只会提交被add到暂存区的内容，没有add，就不会commit

    Branch Management¶

    Create and Merge¶

    • 创建分支 git branch <name>
    • 查看分支 git branch
    • 切换分支 git checkout <name> or git switch <name
    • 创建+切换 git checkout -b <name> or git switch -c <name>
    • 合并到当前分支 git merge <name>
    • 删除分支 git branch -d <name>

    Fix Conflict¶

    何时出现conflict？¶

    1. 你改开头，我改结尾，合并分支，不冲突
    2. 你该开头，我改开头，合并分支，冲突 产生冲突后，Git在产生冲突的文件中用<<<<<<<，=======，>>>>>>>标记出不同分支的内容，需要解决冲突后保存

    解决冲突与合并¶

    将Git标记的内容进行修改，修改后极为解决冲突 合并：再进行 - git add <file> - git commit -m "..." 此时时使用 git log 命令可以看到分支的合并情况 - 带点参数，更清晰的: git log --graph --pretty=oneline --abbrev-commit

    Branch Management Strategy¶

    Bug Branch¶

    [!NOTE] 注意提交 在bug分支上操作要进行提交，如果未提交，直接切换分支，在bug分支中的内容在当前分支中也会有，但是如果此时进行commit，那么bug中的修改会被覆盖，也就是说这次未提交的操作小时了，存在于当前分支了。

    主分支需要修改bug，但是手上工作在dev分支不能停怎么办？ - 使用命令 git stash 不需要进行提交，就可以得到clean working tree 主分支改完回去继续工作？ 1. git stash list 查看刚才的工作现场 2. git stash pop 恢复工作现场的同时把stash内容删除 3. git stash apply stash{0} 恢复指定的工作现场 4. git stash apply 恢复工作现场，不删除 + git stash drop 删除stash内容 在master改动的bug想要同步到dev上？ - git cherry-pick <commit id> commit id 是上次master更改时的id

    Team Work(Remote Branch)¶

    推送分支¶

    推送时要指定本地分支，Git会把该分支推送到远程对应分支 1. git push origin master 2. git push origin dev

    抓取分支¶

    本地git clone远程库之后，默认只有master分支 要在其他分支上开发，需要创建本地分支，和远程分支连接 - git checkout -b dev origin/dev就可以进行正常的提交、push等操作

    解决冲突¶

    两个人对同一个分支进行修改，需要解决冲突 - git pull拉下代码，本地合并、解决冲突，之后推送 - git pull默认得到的是master分支，需要本地先切到master分支 - 要拉取其他分支？ - git branch -set-upstream-to=origin/dev,指定本地dev分支与远程origin/dev分支之间的连接 - git pull 在进行正常的拉取 - 后续解决conflict

    Commands¶

    1. git remote and git remote -v 查看远程库信息
    2. git push origin <branch-name> 推送分支，失败则需先git pull抓取远程的新提交
    3. git checkout -b branch-name origin/branch-name，创建本地对应的分支
    4. git branch --set-upstream <branch-name> origin /<branch-name> 建立本地和远程的的关联

    Git Rease¶

    Tags¶

    tags的意义在于，要找某次提交时，不是死盯着他的commit id，可以直接换成tag

    创建标签¶

    1. 切换到需要打标签的分支
    2. git tag <name> 创建标签
    3. git tag 查看所有标签
    4. 默认在最新的commit上，如果需要打之前的，要git log --...找到commit id，再`git tag
    5. 标签不是俺时间顺序列出的，而是按照字母排序的
    6. git show <tag_name> 查看标签信息
    7. git tag -a <tag_name> -m "comment" <commit_id> 带说明的标签，再git show <tag_name> 可以看到说明

    操作标签¶

    推送标签到远程¶

    1. git push origin <tag_name> 推送一个
    2. git push origin --tags 全部推送

    删除标签¶

    1. 本地删除 git tag -d <tag_name>
    2. 推送到远程之后要删除？
      • 先本地删除
      • 再 git push origin :refs/tags/<tag_name> 远程删除
